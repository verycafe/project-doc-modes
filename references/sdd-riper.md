# SDD-RIPER Reference

Use this reference when the user asks for team vibe coding, spec-driven development, AI coding governance, code execution workflows, or a repository standard that must keep AI work reproducible.

## Core Thesis

Code is cheap; context is expensive. The documentation system should preserve the context needed to restart work, review AI output, and onboard the next agent.

## Three Operating Laws

- No Spec, No Code: code edits need an active requirement, phase plan, and executable spec.
- Spec is Truth: active specs define intended behavior; code that disagrees with approved specs needs review.
- Reverse Sync: bugs and implementation discoveries update specs first or in the same change as code.

If a target repository already says implementation wins over docs, surface the conflict before rewriting governance. For new SDD-RIPER governance generated by this skill, default to active specs as the source of intended behavior.

## State Machine

| Stage | AI Freedom | Required Output | Gate |
| --- | --- | --- | --- |
| Pre-Research | High | CodeMap and context bundle | repository inspected |
| Research | Medium | evidence-backed findings | each claim cites files or docs |
| Innovate | High | 2-3 options with tradeoffs | human chooses direction when multiple viable paths exist |
| Plan | Low | atomic checklist with file paths and acceptance criteria | plan approved before code work |
| Execute | Zero | implementation follows the approved plan | stop if the plan is wrong or incomplete |
| Review | Medium | spec-vs-code verification across function, performance, and security | reviewer accepts or sends back |
| Archive | Low | snapshot, reverse sync, and reusable lessons | current docs stay available |

## Project-Doc-Modes Mapping

- CodeMap: `docs/governance/context/CODEMAP.md`
- Context Bundle: `docs/governance/context/CONTEXT_BUNDLE.md`
- Requirements: `docs/product/vX.Y/requirements/`
- Phase plans: `docs/product/vX.Y/phases/PHASE-*/PLAN.md`
- Executable specs: `docs/product/vX.Y/phases/PHASE-*/specs/`
- Review reports: `docs/product/vX.Y/phases/PHASE-*/REVIEW.md`
- Reverse sync notes: `docs/product/vX.Y/phases/PHASE-*/IMPLEMENTATION_RECORD.md`
- Historical snapshots: `docs/archive/`

Do not create root `specs/` or `changes/` by default. If the user explicitly requests an OpenSpec-style dual directory, place it under `docs/openspec/specs/` and `docs/openspec/changes/` unless the repository already has a different established convention.

## Human-In-The-Loop Gates

Use explicit gates in generated governance docs when code, deployment, or external systems are involved:

- HIL-1 Spec Review: approve requirements, phase plan, and specs before execution.
- HIL-2 Code Review: approve critical code or pull request changes.
- HIL-3 Release Approval: approve production deployment or irreversible operations.
- HIL-4 Reverse Sync Review: confirm implementation records and knowledge updates before archive completion.

For documentation-only scaffolding, HIL-1 can be recorded as pending instead of blocking initial setup.

## Reverse Sync Requirements

Archive is not a trash bin. During archive:

- copy current docs to `docs/archive/`
- keep current docs available under `docs/`
- update implementation records with changed behavior, verification commands, and unresolved risks
- update specs before or with code fixes when a bug exposes missing requirements
- include commit hashes, PR links, deployment links, or issue links only when they exist locally or the user provides them

## Portability Rules

- Treat tool-specific systems such as Claude Code, GitLab, Feishu, Dokploy, OpenSpec, or MCP as optional integrations.
- Keep the portable source of truth in Markdown under `docs/`.
- Generate `.claude/` rules, Git hooks, CI files, or deployment docs only when the user explicitly asks or the target repository already uses them.
